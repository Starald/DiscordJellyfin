# NOTES — лог решений

Короткий журнал «что и почему сделано». Дополняется по ходу работы.

## Архитектурные решения (согласовано до старта)

- **Чистый TypeScript + discord.js** (без NestJS). Бот под один сервер — фреймворк с DI
  только добавил бы boilerplate. ТЗ просит «не переусложнять».
- **Тонкий fetch-клиент Jellyfin** вместо `@jellyfin/sdk`. Используемая часть API маленькая
  (поиск, треки альбома, стрим, картинка) — свой типизированный клиент даёт полный контроль
  и минимум зависимостей.
- **Состояние очереди — в памяти процесса.** Один сервер, без БД. Перезапуск = чистая очередь.
- **Один Discord-сервер, без шардинга.** Команды регистрируются на гильдии (мгновенно).
- **Запуск:** локально Node.js LTS + ffmpeg для разработки; Docker — для деплоя.

## Версии и зависимости

- **Node.js 24.17.0 (LTS)**, ffmpeg 8.1.1 — поставлены через winget.
- **`@discordjs/voice` запинен на `0.19.2`** (точная версия). История DAVE-регрессии:
  - `0.19.0` (08.2025) — ввёл DAVE E2E + voice gateway v8, тут жили баги (циклы реконнекта).
  - `0.19.1` (03.2026) — «Always install Davey», фиксы negative timeouts и одновременных
    переходов состояния (реконнекты).
  - `0.19.2` (03.2026) — strip padding + guards для пакетов. ← берём как «рабочую».
- **`@snazzah/davey` добавлен явной зависимостью** (`^0.1.11`) — в проде сам не подтянется,
  т.к. в voice он лишь dev-зависимость. DAVE сейчас обязателен.
- **Opus: `opusscript` (чистый JS), НЕ `@discordjs/opus`.** Причина: под Node 24 (свежий ABI
  `node-v137`) у `@discordjs/opus@0.10.0` нет prebuilt-бинарника, а собрать из исходников
  нечем (нет Python/VS Build Tools). `opusscript` ставится без компилятора на любом Node.
  Потеря производительности нивелируется тем, что аудио гоним через **FFmpeg → Ogg/Opus
  passthrough**, т.е. opusscript почти не на горячем пути.
- **Шифрование:** нативный `aes-256-gcm` (Node 20+) — есть. `libsodium-wrappers` добавлен
  как fallback. Финальную картину показывает `generateDependencyReport()` на старте.
- **ffmpeg:** системный (winget локально, `apt-get` в Docker), не `ffmpeg-static` — так
  легче образ и совпадает с пожеланием ТЗ ставить FFmpeg в образ.

  командах PATH обновляется из реестра machine+user.

## Веб-панель (ds.starald.ru) — смена курса с Electron

- Пользователь захотел **сайт на поддомене ds.starald.ru**, а не десктоп-приложение.
  Инфраструктура: хостинг — этот Windows-ПК (тут же Jellyfin), прокси — Nginx/NPM,
  доступ — публичный со входом по паролю.
- Архитектура: класс `Bot` (ядро) переиспользован; вместо Electron-IPC — **HTTP-сервер**
  (`src/web/server.ts`, Express 5 + cookie-session). UI — `src/web/ui/` (тот же дизайн).
- Безопасность (публичный сайт): авторизация по логину/паролю (cookie-сессия, подпись
  `SESSION_SECRET`), rate-limit на логин, timing-safe сравнение пароля. Слушает только
  `127.0.0.1` — наружу только через Nginx (TLS). Обложки — через серверный прокси `/art`,
  чтобы **не светить Jellyfin API-ключ** на публичном фронте.
- Запуск: `node dist/index.js` поднимает бота + панель. Деплой — `deploy/DEPLOY.md`
  (DNS, nginx/NPM, TLS, прод-`.env`, автозапуск через NSSM/Планировщик).
- Electron-обёртка осталась в репозитории (`src/electron/`, `npm run app`) как опциональный
  локальный вариант — на веб-панель не влияет.
- Проверено curl: 401 без логина на всех ручках, вход по паролю, /api/* и прокси /art ок.

## Хостинг на ds.starald.ru (выполнено)

- Инфра: всё на этом Windows-ПК. Роутер пробрасывает внешние 80/443 → этот ПК.
- **Caddy** (winget, v2.11) фронтит оба домена с авто-TLS (Let's Encrypt выпущены):
  `starald.ru → 127.0.0.1:8096` (Jellyfin), `ds.starald.ru → 127.0.0.1:8730` (панель).
  Конфиг: `deploy/Caddyfile`.
- Прод-`.env`: `PANEL_SECURE_COOKIE=true`, сгенерён `SESSION_SECRET`, `FFMPEG_PATH`
  (чтобы ffmpeg находился без пользовательского PATH). Пароль пока `test1234` (поменять!).
- **Автозапуск при входе**: VBS в папке «Автозагрузка» (`jellyfin-discord-bot.vbs`,
  копия в `deploy/start-hidden.vbs`) — стартует бот+панель и Caddy скрыто. Я не admin,
  поэтому полноценные службы (старт до входа, авто-рестарт) — опционально через NSSM
  (команды в `deploy/DEPLOY.md`). НЕ использовать VBS и NSSM одновременно (дубли).
- **Гонка `:8000`**: при тестах с этой машины http-редирект отдавал `:8000` даже при
  остановленном Caddy → это перехват локальным **CloudflareWARP** (драйвер активен и в
  «Disconnected»), НЕ проблема деплоя. HTTPS не затронут; внешние клиенты не видят.
  Проверять снаружи (телефон по мобильному интернету).

## Корневая причина «не работает без VPN» (важно!)

- Симптом: без VPN трек «играл» 0.13с и обрывался (плеер: buffering→playing 0.13с→idle по кругу).
  Discord-голос при этом доходил до `ready` — т.е. голос НЕ был заблокирован.
- Причина: бот качал музыку с **публичного** `https://starald.ru`; с этой машины запрос
  наружу ломает WARP/hairpin → FFmpeg получал пустой поток. VPN «чинил» именно скачивание.
- Фикс: `JELLYFIN_URL=http://127.0.0.1:8731` — локальный маршрут Caddy (`:8731` →
  `https://127.0.0.1:8920`, `tls_insecure_skip_verify`, слушает только localhost). Бот берёт
  музыку локально, без публичного крюка → работает без VPN и без zapret. Smoke-тест: 206/audio.
- Диагностика-логи в боте: `[voice …]` и `[player …]` переходы состояния (player.ts).

## YouTube (рабочая схема — важно!)

- Поиск: **YouTube Data API v3** (`YOUTUBE_API_KEY`) — быстро, не зависит от IP, без VPN.
- Воспроизведение: **yt-dlp**, которому нужны (а) чистый IP и (б) свежие куки:
  - `YT_PROXY=http://127.0.0.1:10801` — локальный прокси VPN-клиента **v2RayTun** (Xray).
  - `YT_COOKIES_FROM_BROWSER=firefox` — живые куки из Firefox (Chrome/Edge не читаются:
    app-bound шифрование). Firefox должен быть залогинен в YouTube; экспорт не нужен.
- **КЛЮЧЕВОЕ:** v2RayTun держать в режиме **ПРОКСИ (не ТУННЕЛЬ)**. В режиме туннеля он
  ломает zapret → бот не подключается к Discord (Discord в РФ режется, тянет его zapret).
  В режиме «прокси»: Discord/Jellyfin идут напрямую (zapret), а yt-dlp — через прокси.
- Что не вышло и почему: PO-token провайдер (bgutil) не смог получить BotGuard-challenge
  (виснет) — headless так не умеет; без чистого IP YouTube баннит yt-dlp даже с куки.
- Условия работы YouTube-воспроизведения: запущен v2RayTun (proxy на 10801) + Firefox
  залогинен в YouTube. Если прокси выключить — YouTube не играет, остальное работает.
